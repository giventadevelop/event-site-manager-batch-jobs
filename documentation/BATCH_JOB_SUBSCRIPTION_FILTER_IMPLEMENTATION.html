Batch Job: Subscription Filter Support for Testing v1.0

⚠️ CRITICAL: This is a TESTING/DEVELOPMENT feature only!

  • SCHEDULED JOB UNCHANGED - The batch job that runs every 6 hours remains completely unaffected
  • 7-DAY FILTER PRESERVED - All jobs (scheduled and manual) still only process subscriptions expiring within 7 days
  • TESTING ONLY - When stripeSubscriptionId is provided, filters by specific subscription ID (still validates 7-day renewal)
  • CONDITIONAL LOGIC - Query changes only when testing specific subscription IDs, but renewal date validation always applies

Project Location: E:\project_workspace\event-site-manager-batch-jobs

Purpose: Support filtering by stripeSubscriptionId for testing specific subscriptions, while preserving existing scheduled job behavior and maintaining 7-day renewal date validation

1. Overview

1.1 Current Situation

The batch job query only processes subscriptions expiring within 7 days. When testing specific subscriptions, we need to be able to target a specific subscription ID, but the 7-day renewal date validation is still enforced to ensure only subscriptions approaching renewal are processed.

Current Batch Job Log:

Received request to trigger subscription renewal job for tenant: tenant_demo_002,
stripeSubscriptionId: null

Loaded 0 subscriptions for renewal processing

1.2 What We're Adding

  • Conditional query logic - If stripeSubscriptionId is provided, filter by it and process only that subscription
  • Preserve scheduled job behavior - If stripeSubscriptionId is NOT provided, use existing 7-day filter
  • Repository method support - Added findByStripeSubscriptionIdAndTenantId method to repository
  • Reader configuration - Reader accepts stripeSubscriptionId via setter method

1.3 What Remains Unchanged

✅ The scheduled batch job (runs every 6 hours) is completely unaffected:

  • Scheduled jobs don't provide stripeSubscriptionId parameter (passes null)
  • Reader uses existing 7-day filter logic (unchanged behavior)
  • Still processes subscriptions expiring within 7 days
  • No performance impact on scheduled runs
  • When stripeSubscriptionId is provided, renewal date check is still performed (subscription must be approaching renewal)

Java - Key Changes in SubscriptionRenewalReader
// Added field to store stripeSubscriptionId
private String stripeSubscriptionId;

// Added setter method
public void setStripeSubscriptionId(String stripeSubscriptionId) {
    this.stripeSubscriptionId = stripeSubscriptionId;
    this.subscriptionIterator = null; // Reset iterator
}

// Modified read() method with conditional logic
@Override
public MembershipSubscription read() throws Exception {
    if (subscriptionIterator == null) {
        LocalDate checkDate = LocalDate.now().plusDays(daysBeforeRenewal);
        List<MembershipSubscription> subscriptions;

        // ✅ KEY CHANGE: If stripeSubscriptionId is provided, process only that subscription
        if (stripeSubscriptionId != null && !stripeSubscriptionId.isEmpty()) {
            if (currentTenantId != null && !currentTenantId.isEmpty()) {
                Optional<MembershipSubscription> subscription = repository
                    .findByStripeSubscriptionIdAndTenantId(stripeSubscriptionId, currentTenantId);

                if (subscription.isPresent()) {
                    MembershipSubscription sub = subscription.get();
                    // Still check if it's approaching renewal (7-day filter still applies)
                    if (sub.getCurrentPeriodEnd() != null &&
                        (sub.getCurrentPeriodEnd().isBefore(checkDate) ||
                         sub.getCurrentPeriodEnd().isEqual(checkDate)) &&
                        !Boolean.TRUE.equals(sub.getCancelAtPeriodEnd()) &&
                        "ACTIVE".equals(sub.getSubscriptionStatus())) {
                        subscriptions = Collections.singletonList(sub);
                    } else {
                        subscriptions = Collections.emptyList();
                    }
                } else {
                    subscriptions = Collections.emptyList();
                }
            } else {
                // Fallback: find by stripeSubscriptionId only
                MembershipSubscription subscription = repository
                    .findByStripeSubscriptionId(stripeSubscriptionId);
                // ... similar validation logic ...
            }
        } else if (currentTenantId != null && !currentTenantId.isEmpty()) {
            // Existing logic: Process all subscriptions for tenant with 7-day filter
            subscriptions = repository.findByTenantIdAndSubscriptionStatus(currentTenantId, "ACTIVE");
            subscriptions = subscriptions.stream()
                .filter(s -> s.getCurrentPeriodEnd() != null &&
                    (s.getCurrentPeriodEnd().isBefore(checkDate) ||
                     s.getCurrentPeriodEnd().isEqual(checkDate)))
                .filter(s -> !Boolean.TRUE.equals(s.getCancelAtPeriodEnd()))
                .filter(s -> s.getStripeSubscriptionId() != null &&
                    !s.getStripeSubscriptionId().isEmpty())
                .limit(maxSubscriptions)
                .toList();
        } else {
            // Existing logic: Process all subscriptions with 7-day filter
            subscriptions = repository.findSubscriptionsApproachingRenewal(checkDate);
            // ... limit logic ...
        }

        subscriptionIterator = subscriptions.iterator();
        log.info("Loaded {} subscriptions for renewal processing", subscriptions.size());
    }

    if (subscriptionIterator.hasNext()) {
        return subscriptionIterator.next();
    }
    return null;
}

7. Important Notes

⚠️ Performance Considerations:

  • When stripeSubscriptionId is provided, the query uses findByStripeSubscriptionIdAndTenantId which should use indexes on both fields (fast lookup)
  • When stripeSubscriptionId is NOT provided, the query uses existing repository methods with indexes on current_period_end (unchanged performance)
  • No performance impact on scheduled jobs
  • Single subscription lookup is very efficient (O(1) with proper indexes)

✅ Backward Compatibility:

  • Existing scheduled jobs continue to work unchanged
  • Manual triggers without stripeSubscriptionId behave the same as scheduled jobs
  • Only manual triggers WITH stripeSubscriptionId get new behavior

8. Summary

  • ✅ Preserving 7-day filter for scheduled jobs
  • ✅ Maintaining backward compatibility

What we're NOT doing:

  • ❌ Not changing scheduled job behavior

  • ❌ Not affecting production batch job runs
  • ❌ Not creating new batch job types
  • ❌ Not bypassing renewal date validation

Key Design Principle:

"When stripeSubscriptionId is provided → Testing mode (filter by specific subscription ID, still validate 7-day renewal)

When stripeSubscriptionId is NOT provided → Production mode (use 7-day filter for all subscriptions)"

Important: The 7-day renewal date check is always applied - subscriptions must be approaching renewal to be processed, regardless of whether stripeSubscriptionId is provided.
